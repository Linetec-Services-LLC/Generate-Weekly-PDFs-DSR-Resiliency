# Quick Task 260601-k34 — SUMMARY

## What shipped

`portal-v2/src/components/auth/ResetPasswordPage.tsx` now handles the Supabase
**token_hash (PKCE) recovery flow** in addition to the implicit
`PASSWORD_RECOVERY` event. This was the only remaining repo code change for
Phase 04 plan 04-06, item C.

### Root cause it fixes
Once the Supabase "Reset Password" email template is corrected to link to
`{{ .RedirectTo }}?token_hash=...&type=recovery` (item A), the recovery link
uses the token_hash flow, which does **not** emit a `PASSWORD_RECOVERY`
`onAuthStateChange` event. The old component only listened for that event, so
the page hung forever on "Verifying your reset link…". The form never unlocked.

### Change
In the mount `useEffect`:
- Read `token_hash` and `type` from `window.location.search`.
- If `type === 'recovery'` and `token_hash` is present, call
  `supabase.auth.verifyOtp({ token_hash, type: 'recovery' })`; on success set
  `readyToReset(true)`, on error set the existing
  "This reset link has expired. Request a new one." copy.
- Keep the existing `PASSWORD_RECOVERY` `onAuthStateChange` listener as the
  implicit-flow fallback (no regression for older links).

## Tests

New file `portal-v2/src/components/auth/__tests__/ResetPasswordPage.test.tsx`
(first test for this component), 5 cases — written RED-first, then GREEN:
1. token_hash recovery link → `verifyOtp({ token_hash, type: 'recovery' })`.
2. verifyOtp success → form enabled ("Enter your new password below.").
3. verifyOtp error → "This reset link has expired…" surfaced.
4. No token_hash → `verifyOtp` not called (implicit flow preserved).
5. `PASSWORD_RECOVERY` event → form still enabled (fallback intact).

framer-motion is mocked to plain elements (strips animation-only props) for
deterministic jsdom rendering; supabase/useToast/react-router-dom mocked per the
existing `AuthGuard.test.tsx` convention.
