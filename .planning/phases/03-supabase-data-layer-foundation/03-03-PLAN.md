<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions secrets → publish step env | The service_role JWT crosses into the step env; it must stay in CI/Supabase only, never on Vercel/frontend. |
| publish step exit code → billing job result + cache/hash_history | A publish failure must be isolated so it cannot fail the billing run or block `if: always()` cache-save steps. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-secret-scope | Information Disclosure / Elevation | service_role key in workflow | mitigate | Step reuses `secrets.SUPABASE_SERVICE_ROLE_KEY` (CI-only); verify gate asserts no `VITE_` var on the step; the key is never set on Vercel (enforced milestone-wide in Phase 07 grep audit). |
| T-03-publish-dos | Denial of Service (operational) | publish step exit code | mitigate | `continue-on-error: true` + placement BEFORE the `if: always()` cache-save block — a Supabase outage cannot fail the run or lose `hash_history.json` (Pitfall C); ordering enforced by the verify gate. |
| T-03-step-leak | Information Disclosure | step logs | accept | The script (Plan 02) logs only aggregate counts + exception types; GitHub masks registered secrets in step logs. No per-file PII is emitted by the step itself. |
</threat_model>
