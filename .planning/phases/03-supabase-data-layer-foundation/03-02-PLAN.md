<objective>
Build the additive CI publish script `scripts/publish_artifacts_to_supabase.py`
that scans the run's generated Excel files, uploads each to the private
`excel-artifacts` Storage bucket, and upserts an idempotent `public.artifacts`
metadata row keyed on `sha256` — wrapped so a Supabase outage is LOUD but never
fatal (D-06). This is TDD: the mocked test module (Wave 0) is written first and
must fail RED, then the script is implemented to GREEN.

(D-05) — `generate_weekly_pdfs.py` is NOT touched. It reuses the battle-tested
`billing_audit.client` retry/circuit-breaker and the manifest module's
file-hash helper so the additive script inherits the established Supabase
posture (no hand-rolled retry, no hand-rolled SHA256).

Output: `tests/test_publish_artifacts_to_supabase.py` (fully mocked — no real
Supabase calls, no real uploads) and `scripts/publish_artifacts_to_supabase.py`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI runner (service_role) → Supabase Storage + Postgres | The publish step writes with the service_role JWT (bypasses RLS); the key must never be logged or leaked. |
| Excel filename tokens → upsert payload / SQL | Filenames embed WR/foreman/customer names (PII) and are untrusted-shaped input to the row. |
| publish step exit code → billing job result | The publish process must not be able to fail the billing run. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-secret | Information Disclosure | service_role key | mitigate | Key read from `os.getenv` inside `get_client()`; never printed/logged; `test_secret_not_logged` asserts the value is absent from log output. |
| T-03-pii-sentry | Information Disclosure | Sentry/log bodies | mitigate | Failure logs emit only `type(exc).__name__` + aggregate count; never the raw filename (Pitfall D); `test_no_pii_in_sentry_body` enforces it. |
| T-03-publish-dos | Denial of Service (operational) | publish process exit code | mitigate | Internal try/except -> Sentry -> `$GITHUB_STEP_SUMMARY` -> exit 0 (defense in depth atop Plan 03's `continue-on-error: true`); `test_failure_isolation` enforces no raise / no non-zero exit. |
| T-03-sqli | Tampering | upsert payload | mitigate | supabase-py parameterizes the upsert dict; no string-built SQL anywhere. |
| T-03-variant-coerce | Tampering / data loss | `variant` value | mitigate | Unknown token captured to Sentry (type+count, no PII) and still inserted loudly — no silent coercion, no DB CHECK to hard-drop it. |
</threat_model>
