> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>

> primitives. Deep visual polish and Framer Motion animations are explicitly

## Role Model Contract

**CRITICAL:** The authoritative role taxonomy is `admin | billing | pending`
(from `supabase/portal_schema.sql`, Phase 03 D-11). The existing
`portal-v2/src/lib/types.ts` is STALE (`admin | viewer | biller`).

Phase 04 must reconcile:

| Stale (types.ts today) | Authoritative (schema) | Action |
|------------------------|------------------------|--------|
| `viewer` | — | DROP — no equivalent, no UI references |
| `biller` | `billing` | RENAME — update `UserRole`, `ROLES` array, all Badge labels |
| `admin` | `admin` | KEEP — no change |
| — | `pending` | ADD — new role; drives approval-screen routing |

Updated `UserRole` type: `'admin' | 'billing' | 'pending'`

Updated `Profile` interface:
```ts
interface Profile {
  id: string;
  email: string;          // populated by handle_new_user() trigger
  role: UserRole;         // 'admin' | 'billing' | 'pending'
  created_at: string;     // populated by trigger
}
```
Fields to DROP from Profile: `display_name`, `is_active`, `updated_at`
(not present in the deployed schema; their removal is part of reconciliation).

## Spacing Scale

Declared values — multiples of 4 only. Pre-populated from existing component
patterns observed in LoginPage.tsx, UsersPage.tsx, and Toast.tsx.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps (e.g. `gap-1`), badge inner padding |
| sm | 8px | Compact row padding, inline form element gaps |
| md | 16px | Default form field spacing, card internal sections |
| lg | 24px | Card padding (`p-6`), section breaks between form groups |
| xl | 32px | Page-level top/bottom padding |
| 2xl | 48px | Auth page vertical centering buffer |
| 3xl | 64px | Not used in Phase 04 |

Exceptions:
- Touch targets for icon-only buttons (show/hide password, dismiss toast): minimum
  44px hit area via `p-2` + icon `size={16}` (matches existing LoginPage pattern).
- hCaptcha widget: renders at its native height (~78px); do not constrain with
  `max-h` or `overflow-hidden` on the containing element.

## Typography

Pre-populated from LoginPage.tsx, UsersPage.tsx patterns.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px (`text-sm`) | 400 (regular) | 1.5 | Form labels, table cells, descriptive copy |
| Label | 14px (`text-sm`) | 400 (regular) | 1.4 | Form field labels, "Remember me" checkbox label |
| Heading | 20px (`text-xl`) | 700 (bold) | 1.2 | Page headings (Users, Pending Approval) |
| Display | 24px (`text-2xl`) | 700 (bold) | 1.2 | Auth card title ("Linetec Portal") |

**2 declared weights (400, 700), plus one inherited `font-semibold` exception
on table headers.** The two weights are **400** (body copy, form labels,
placeholder, secondary text) and **700** (headings, display, CTAs).

Why labels are 400 (not 500): existing form labels used `font-medium` (500) in
LoginPage.tsx, but at 14px on the system-UI stack 500 is visually
indistinguishable from 400, so the contract collapses labels to **400** and
removes weight 500 from the declared set. New auth surfaces use no weight class
(`font-normal`) on labels. The pre-existing `font-medium` on LoginPage labels
may remain untouched since it renders identically — but do not introduce new
500-weight text anywhere.

Inherited exception (NOT a third declared weight): the table-header uppercase
labels in UsersPage.tsx use `font-semibold` (600) via the existing
`text-xs font-semibold uppercase tracking-wide` pattern. This 600 weight is
carried verbatim from working markup and is left untouched — rewriting the
existing header is out of scope for this phase. It is explicitly an inherited
exception, not part of the 2-weight declared contract.

## Color

Pre-populated from tailwind.config.ts tokens and existing component patterns.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `slate-950` / `slate-900` gradient | Auth page full-screen background; `from-slate-950 via-slate-900 to-red-950` |
| Secondary (30%) | `bg-white/10` (glass) on auth; `bg-white` on admin surfaces | GlassCard panels on auth pages; white card panels on admin pages (UsersPage pattern) |
| Accent (10%) | `#C41230` (`brand-red`) | Primary CTA buttons only; avatar initials background; focus ring (`ring-brand-red/60`) |
| Destructive | `red-500` / `red-300` | Error inline messages, error toast border (`border-l-red-500`), "last admin" warning banner |

Accent reserved for: **primary submit buttons only** (Sign In, Create Account,
Send Reset Email, Set New Password, Save Role Change). Do not apply `brand-red`
to links, secondary actions, or informational badges.

Role-badge color mapping (using existing Badge variants):

| Role | Badge variant | Visual |
|------|---------------|--------|
| `admin` | `info` | blue — `bg-blue-100 text-blue-700` |
| `billing` | `success` | emerald — `bg-emerald-100 text-emerald-700` |
| `pending` | `warning` | amber — `bg-amber-100 text-amber-700` |

Pending users MUST visually stand out in the Users table. Use `warning` badge
AND a subtle amber row tint (`bg-amber-50/40`) so admins can scan for pending
users at a glance without a separate filter.

Config-error surface (D-12 / D-13): full-screen centered layout with
`bg-slate-900` background, `GlassCard` container, `XCircle` icon in `text-red-400`,
white heading, `text-white/60` body copy. This is distinct from the auth pages —
it must render before the router initializes.

### 5. UsersPage (modified)

**File:** `portal-v2/src/components/admin/UsersPage.tsx`
**Route:** `/admin/users`
**Access:** `admin` role only (via `RoleGuard allow={['admin']}` + DB RLS)

Layout (existing table structure — modify, do not rebuild):

Table columns (updated from existing):

| Column | Data | Notes |
|--------|------|-------|
| User | avatar initial + email | Drop `display_name` — show `email` only; avatar initial from `email[0].toUpperCase()` |
| Role | `<select>` (admin / billing / pending) | Replace `viewer/biller` with `billing/pending`; last-admin guard disables own-role select |
| Status | Badge | Use role-based badge coloring (admin=info, billing=success, pending=warning); replaces `is_active` boolean |
| Joined | `created_at` formatted | Keep as-is |

Pending-user highlight: rows where `role === 'pending'` receive
`className="... bg-amber-50/40"` in addition to the existing hover style.
This is additive — do not remove the `hover:bg-slate-50/50 transition-colors`.

Pending count badge: in the page heading area, if any pending users exist,
show a `warning` Badge next to the "Users" heading:
`{n} pending` — e.g. `<Badge variant="warning">2 pending</Badge>`.

Last-admin guard:
- When the authenticated user is the only admin, their own role `<select>`
  is `disabled` with `opacity-50 cursor-not-allowed`.
- Additionally, attempting to change another user's role TO a non-admin
  value when that user is the last admin is blocked: show a `warning` toast:
  "Cannot demote the last admin. Promote another user to admin first."
- No modal confirmation needed — the disabled state and toast are sufficient.

Error state (replaces the bare `<p className="text-sm text-red-500">`):
Render an inline error card using `GlassCard`-style or bordered container:
`bg-red-50 border border-red-200 rounded-xl p-4 text-sm text-red-700` with
a retry button.

Empty state (no users yet, unlikely but must exist):
`text-slate-400 text-sm text-center py-12` — "No users found."

### 6. AuthGuard + RoleGuard (modified / new)

**File:** `portal-v2/src/components/auth/AuthGuard.tsx` (modify)
**New file:** `portal-v2/src/components/auth/RoleGuard.tsx` (new)

- 403 inline view: centered, `text-slate-500 text-sm`, "You don't have
  permission to view this page." with a "Go to dashboard" link

Config-error surface (for `supabase.ts` missing-env fail-loud):
- Render BEFORE router/auth initialization
- Full-screen `bg-slate-900`, centered GlassCard, `XCircle` in

## Accessibility Contract

| Requirement | Implementation |
|-------------|---------------|
| All form inputs have associated `<label>` | Use `<label htmlFor={id}>` + matching `id` on every `<input>` and `<select>` |
| hCaptcha widget is keyboard-accessible | Use `@hcaptcha/react-hcaptcha` default checkbox variant — it is natively keyboard-focusable |
| Password show/hide button | `aria-label={showPassword ? 'Hide password' : 'Show password'}` (existing pattern) |
| Role `<select>` when disabled | `aria-disabled="true"` + `title="You are the last admin and cannot change your own role"` |
| Error messages programmatically associated | `aria-describedby` linking input to its error element ID |
| Color contrast | White text on `slate-900`/`slate-950` backgrounds: WCAG AA (>7:1). `brand-red` (#C41230) on white: verify ≥4.5:1 for body text; use on buttons only (white label text on red bg passes). `amber-700` on `amber-100`: ≥4.5:1 (meets AA). |
| Focus ring | `focus:ring-2 focus:ring-brand-red/60` (existing pattern); do not suppress focus outlines |
| Toast dismiss | `aria-label="Dismiss"` on dismiss button (existing Toast.tsx pattern) |
| PendingApprovalPage | No nav links; `role` attribute not needed; all text has sufficient contrast |
| Skip-to-main | Not required in Phase 04 — single-purpose pages (auth forms, single-panel admin) |
