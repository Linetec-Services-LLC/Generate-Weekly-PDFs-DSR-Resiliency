<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| anonymous browser → auth forms | login/signup/forgot are unauthenticated entry points; bot/abuse risk |
| recovery email link → /auth/reset | the URL fragment carries a recovery token |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-13 | Spoofing | credential stuffing / brute force on login + signup | mitigate | hCaptcha widget gates both forms; token passed via `options.captchaToken`; submit disabled until token present (Task 1). Supabase validates the token server-side |
| T-04-14 | Spoofing | captcha token replay / stale token | mitigate | Reset the widget (`resetCaptcha()` + clear token) in the `finally` of every submit (Tasks 1, 2) — tokens are single-use and ~2-min lived (Pitfall 1) |
| T-04-15 | Information Disclosure | user-enumeration via forgot-password | mitigate | ForgotPasswordPage always shows the success card on the happy path; never reveals whether the email exists (Task 2, UI-SPEC) |
| T-04-16 | Spoofing | password-reset session not established before updateUser | mitigate | ResetPasswordPage gates the submit on `readyToReset` set by the PASSWORD_RECOVERY event (Task 3, Pitfall 3) — prevents "Auth session missing" |
| T-04-17 | Elevation of Privilege | pending user bypassing the approval screen | mitigate | signup navigates to /pending (Task 1); AuthGuard (Plan 03) independently re-routes any pending session away from the dashboard |
</threat_model>
